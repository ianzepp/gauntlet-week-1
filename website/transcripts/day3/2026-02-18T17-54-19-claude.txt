## 4. Missing Font Loading

The React global.css references two custom web fonts that are not loaded anywhere in Leptos:

- **"IBM Plex Mono"** â€” used as `--font-mono` for virtually every text element in the UI (labels, inputs, buttons, metadata, toolbars, status bar, inspector, chat, AI panel). Requires a `<link>` to Google Fonts or a local `@font-face`.
- **"Courier Prime"** â€” fallback in `--font-mono` stack
- **"JetBrains Mono"** â€” second fallback in `--font-mono` stack
- **"Caveat"** â€” used as `--font-script` for board names (BoardCard, BoardStamp title, DashboardPage dialog input, UserFieldReport name). A Google Fonts handwriting font that gives the notebook/sketchpad feel.

Leptos `main.css` has no `@import`, no `@font-face`, and no `<link>` to any font CDN. The body falls back to system-sans, losing the entire typographic identity of the React design.

## 5. CSS Module Naming vs. BEM â€” Adaptation Pattern

React uses CSS Modules where locally-scoped class names are generated at build time. In JSX:
```jsx
import styles from './Toolbar.module.css';
<div className={styles.toolbar}>
  <div className={styles.left}>
    <span className={`${styles.presenceChip} ${isActive ? styles.active : ''}`}>
```

The local class names (`toolbar`, `left`, `presenceChip`, `active`) are unambiguous within the module file but become globally unique hashes at runtime.

Leptos uses plain BEM in a single `main.css` file. The equivalent pattern is:
```
.toolbar                   â†’ .toolbar (block)
.toolbar .left             â†’ .toolbar__left (element)
.presenceChip + active     â†’ .toolbar__presence-chip--active (modifier)
```

Key translation issues:

1. **Compound class conditionals** â€” React: `className={[styles.toolButton, active && styles.active].filter(Boolean).join(' ')}`. Leptos: attribute `class="tool-rail__btn tool-rail__btn--active"`. The BEM modifier must be added as a separate class, not replaced.

2. **Nested selectors that rely on co-location** â€” React `.mini .cardName { font-family: var(--font-mono); }` (a context override within the module). In BEM this becomes `.board-card--mini .board-card__name { }` â€” the nesting is explicit but the rule must exist in the single file at the right scope.

3. **State classes** â€” React's `.active`, `.disabled`, `.connected`, `.offline` are all module-local (no collision risk). In Leptos BEM they need full qualification: `.tool-rail__btn--active`, `.status-bar__dot--connected`. The Leptos file currently uses shorthand modifiers like `--connected`, `--connecting`, `--disconnected` on the dot which is correct BEM, but several places use simple state classes (`.board-card:hover`, `.left-panel--collapsed .left-panel__content`) without proper BEM modifier equivalents.

4. **The `stripAnchor` (LeftPanel)** â€” React uses `position: fixed; left: 52px` which depends on the rail being exactly 52px. The BEM class `.left-panel__strip-anchor` would need explicit documentation of this positional dependency, which is currently absent in Leptos.
