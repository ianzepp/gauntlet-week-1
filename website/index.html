PROJECT / IDENTIFICATION
CollabBoard
Ian · Rust · Leptos · Axum · 7 Days
DISCOVERY_DESCRIPTION / SUMMARY

A real-time collaborative whiteboard where multiple users draw, chat, and let AI rearrange things on a shared canvas — built entirely in Rust, from the server all the way down to the browser.

Open a board, grab a tool, start sketching. Rectangles, ellipses, diamonds, stars, lines, arrows, text elements, frames that group objects, and embedded YouTube tiles. Other users on the same board see every stroke as it happens. Type a prompt and the AI reads your board state, decides which tools to call, and mutates objects directly — you watch it happen live.

Built in 7 days for The Gauntlet. The twist: do it all in Rust — server, frontend, canvas engine, wire protocol — no JavaScript runtime anywhere in the stack.

SYSTEM_ARCHITECTURE

Six Rust crates in a Cargo workspace. The client compiles to WASM and runs in the browser via Leptos SSR + hydration. The canvas engine has zero browser dependencies in its core — it compiles to native Rust for testing and to WASM for the browser.

Communication uses binary protobuf frames over WebSocket. The server routes by syscall prefix — board:*, object:*, cursor:*, chat:*, ai:* — and handlers return an Outcome enum. A single dispatch layer decides who gets what.

Two-speed persistence: object dirty flush at 100ms intervals, frame log queue with batched writer at 5ms. AI integration runs a tool-call loop with 9 tools that read and mutate the board directly.

CRATE_INVENTORY
SERVER — THE BACKEND 14,006 LOC

Axum HTTP server, WebSocket hub, and persistence layer. Handler functions return an Outcome enum and a single dispatch layer decides routing. Two-speed persistence with object dirty flush and frame log queue.

CLIENT — THE FRONTEND 15,914 LOC

Leptos 0.8 application that renders on the server (SSR) and hydrates in the browser as a WASM binary. Pure Rust, no TypeScript, no JS framework. Eight RwSignal contexts for state management.

CANVAS — THE ENGINE 8,897 LOC

From-scratch 2D whiteboard engine. Compiles to native Rust for testing, compiles to WASM for the browser. Zero browser dependencies in the core. Seven gesture states, nine shape kinds, full hit testing.

FRAMES — THE WIRE PROTOCOL 610 LOC

Shared crate defining the Frame type and encode/decode functions. Binary protobuf via Prost 0.13. Status lifecycle: Request → Item* → Done | Error | Cancel.

TRACES — OBSERVABILITY 1,453 LOC

Shared trace and event primitives for the observability UI. Syscall prefix-to-display mapping, trace filtering, frame-to-session grouping, waterfall timing, aggregate metrics.

PERF — THE LOAD HARNESS 698 LOC

End-to-end performance tests over real HTTP and WebSocket connections. Round-trip latency, board complexity scaling, and 25-user concurrent load scenarios.

TECHNOLOGY_STACK

The entire stack is Rust. The server is Axum with SQLx for database access. The frontend is Leptos — server-side rendered, then hydrated in the browser as WASM. No JavaScript runtime anywhere.

Auth supports GitHub OAuth and email access codes with single-use WebSocket tickets. AI features work with either Anthropic (Claude) or OpenAI-compatible backends. Deployed on Railway with Docker, migrations run automatically on startup.

DESIGN_DECISIONS
WHY_RUST_END_TO_END:

"No JavaScript runtime anywhere in the stack. Server, frontend, canvas engine, wire protocol — one language, one type system, shared crates between client and server."

WHY_PROTOBUF:

"Binary wire format for WebSocket frames. Compact, typed, with a recursive serde_json::Value to prost_types::Value conversion for flexible data payloads."

WHY_OUTCOME_ENUM:

"Handlers return an Outcome — Broadcast, Reply, ReplyStream — and dispatch decides routing. Keeps handlers pure and testable. They never touch the socket directly."

WHY_TWO_SPEED_PERSISTENCE:

"Object dirty flush at 100ms, frame log queue with batched writer at 5ms. Each path tuned to its traffic pattern. Ephemeral frames never hit the database."

WHY_NATIVE_CANVAS_ENGINE:

"EngineCore has zero web-sys dependencies. The full test suite runs in cargo test without a browser. The WASM wrapper just forwards DOM events and calls render()."

DEMONSTRATION_GUIDE

The live demo runs on Railway. Log in with GitHub or an email access code, create a board, and start drawing. Open the same board in a second browser tab to see real-time sync.

Try the AI: click the AI button and type a natural language instruction like "arrange these sticky notes in a 3x3 grid" or "add a red diamond labeled URGENT." The AI reads your board, picks tools, and mutates objects live.

Other features to explore: follow-camera (click another user's avatar to sync your viewport), chat (persistent per-board messaging), savepoints (rewind the board to an earlier state), and the compass widget (rotate the viewport).

FEATURE_INVENTORY
DRAWING TOOLS

Rectangles, ellipses, diamonds, stars, lines, arrows, text, frames (grouping), YouTube embeds. Full resize, rotate, multi-select.

REAL-TIME SYNC

WebSocket-based. Every object mutation broadcasts to all peers instantly. Cursor presence with deadband filtering. Smooth lerp on incoming drag events.

AI INTEGRATION

Natural language board manipulation. 9 tools: create sticky notes, shapes, frames, connectors; move, resize, update text, change color, get board state.

FOLLOW CAMERA

Click another user's avatar to sync your viewport to theirs. Pan, zoom, and rotation all sync. Click again to detach.

CHAT & PRESENCE

Persistent per-board chat. User presence with live cursor positions. Heartbeat-based connection status.

SAVEPOINTS

Snapshot the board state at any point. Rewind to a previous savepoint to undo bulk changes.
